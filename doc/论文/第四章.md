## 4 基于 FPGA 的 TCC-SOCS 空中像计算加速架构设计与实现

### 4.1 FPGA 加速方案总体设计

第 3 章通过 MATLAB 仿真验证了“离线预计算 TCC + 在线 SOCS 重构”算法的可行性，结果表明采用 \(N_k = 10\) 个本征核即可实现 SSIM = 0.9966 的高保真空中像重构。然而，工业级 OPC（光学邻近效应校正）流程需对大规模版图中的海量切割窗口进行反复成像评估，而单次 SOCS 空间像计算在 CPU 平台上仍需约 0.287 秒（含 FFTW 调用），难以满足实时性要求。本节针对 SOCS 算法固有的并行性与流水线特性，提出了一种基于 Xilinx Kintex UltraScale+ FPGA 平台的软硬件协同加速方案。

#### 4.1.1 硬件加速需求与目标

SOCS 在线成像的核心计算可表述为：

$$
I(x,y) = \sum_{k=1}^{N_k} \sigma_k \left| \mathcal{F}^{-1}\{K_k(\nu_x,\nu_y) \cdot M(\nu_x,\nu_y)\} \right|^2
$$

其中，\(\mathcal{F}^{-1}\) 表示二维逆傅里叶变换，\(K_k\) 为第 \(k\) 个本征核的频域表示，\(M\) 为掩模频谱。该公式揭示了三个层次的计算密集型操作：

1. **频域点乘**：\(K_k \cdot M\) —— 每个核需对 \(k_x \times k_y\) 个频域采样点执行复数乘法；
2. **二维 IFFT**：\(128 \times 128\) 点逆变换 —— 算法计算复杂度的主体，复杂度为 \(O(N^2 \log N)\)；
3. **强度累加**：\(|E|^2 \times \sigma_k\) —— 逐像素平方求和与加权累加。

上述三个操作在 \(N_k\) 个核之间完全独立，天然适合并行化；而二维 IFFT 内部的行-列分解结构又为流水线化提供了有利条件。这些特性使得 FPGA 成为加速 SOCS 计算的理想硬件平台。

表 4-1 给出了本设计的核心加速目标与实测性能指标。

**表 4-1 SOCS 计算核心加速指标**

| 指标               | 目标值           | 实测值                                                    | 备注                         |
| ------------------ | ---------------- | --------------------------------------------------------- | ---------------------------- |
| 时钟频率           | \(\geq 200\) MHz | 250–280 MHz（预估，待时序收敛）                           | 关键路径见 §4.5.3            |
| 单次 SOCS 计算延迟 | \(< 30\) ms      | C 综合：\(\approx 9.62\) ms @ 274 MHz（2,636,242 周期）；Co-Sim：2,651,856 周期 | RTL 仿真含接口开销           |
| 计算精度 (RMSE)    | \(< 10^{-5}\)    | \(2.93 \times 10^{-8}\)                                   | 远优于设计目标               |
| BRAM 占用率        | \(< 75\%\)       | 55%（xcku3p）/ 42%（xcku5p）                              | 基于 V18 综合报告            |

基于 SOCS 算法的计算特性，本设计采用主机预处理与 FPGA 核心计算相结合的异构协同架构。划分原则为：涉及高精度浮点矩阵运算与特征值分解的离线计算由 CPU 完成，而具有高度并行性与规则数据访问模式的在线成像计算则由 FPGA 承担。

**数据流概览**：输入数据从主机 DDR 经 AXI-MM Master 接口以突发模式读取至 FPGA 内部 BRAM 缓冲区，经过核心计算流水线处理后，结果经 AXI-MM 接口写回 DDR。主机通过 AXI-Lite 接口动态配置核数量 \(n_k\)、光学参数 \(N_x\)、\(N_y\) 以及掩模尺寸 \(L_x\)、\(L_y\)，并控制 IP 核的启动与停止。

核心计算流程可简述如下（伪代码）：

```pseudocode
// 初始化累加缓冲区
tmpImg[128][128] ← 0

// 逐核迭代 (nk = 10)
for k = 0 to nk-1 do
    // 阶段 1: 频域嵌入 —— K_k × M 点乘 + 零填充
    fft_input ← embed(krn_r[k], krn_i[k], mskf_r, mskf_i)
    
    // 阶段 2: 二维 IFFT（行-列分解）
    fft_output ← IFFT2D(fft_input)
    
    // 阶段 3: 加权强度累加
    tmpImg ← tmpImg + σ_k × |fft_output|²
end for

// 阶段 4: 频移居中
tmpImgp ← fftshift(tmpImg)

// 阶段 5: 写回 DDR
output_ddr ← tmpImgp
```

### 4.2 计算核心的并行化设计

#### 4.2.1 SOCS 算法的并行性分析

SOCS 算法的核心公式为 \(I = \sum_{k=1}^{N_k} \sigma_k |E \otimes K_k|^2\)。其中，各核的计算路径完全独立——每个核独立完成“频域点乘 → IFFT → 强度计算”流程，仅在最终的累加步骤存在数据依赖。这种结构天然蕴含两种并行性。

**1. 核间并行性（Kernel-level Parallelism）**：\(N_k = 10\) 个核的计算相互独立，理论上可分配至 10 条独立流水线并行执行，可获得约 10× 的加速比。然而，每个核需要一套完整的 \(128 \times 128\) IFFT 引擎及相应临时缓冲区，全并行实现将消耗约 \(10 \times 304 = 3040\) 个 BRAM，远超目标器件容量。因此，在资源受限条件下，设计中采用时分复用策略，通过循环迭代方式复用同一套计算单元。

**2. IFFT 内部流水性（Pipeline Parallelism）**：二维 IFFT 采用行-列分解法，128 行变换可逐行流水执行，行变换完成后逐列执行列变换。每行/列的 128 点 FFT 内部又可进一步分解为 \(\log_2 128 = 7\) 级蝶形运算流水线。这种层次化的流水结构使得 IFFT 成为典型的流水线友好型计算任务。

鉴于目标器件 xcku5p-ffvb676-2-e 的 BRAM 容量限制（960 个 BRAM_18K），本设计最终采用**核间时分复用 + 核内子函数流水**的混合并行策略。在顶层循环中，10 个核顺序执行，各阶段之间采用顺序调用模式而非任务级 DATAFLOW。该策略确保了 C/RTL 联合仿真的确定性延迟推断，同时各子函数内部通过 `PIPELINE II=1` 指令实现了指令级流水，从而在资源约束下维持了较高的计算吞吐率。

#### 4.2.2 计算流程的阶段分解

整个 SOCS 计算流程可分解为五个相互独立的功能阶段，每个阶段对应一个专用功能模块。表 4-4 列出了各阶段的主要功能、关键操作及数据规模。

**表 4-4 SOCS 计算流程阶段分解**

| 阶段    | 功能      | 关键操作                                     | 数据规模                              |
| ------- | --------- | -------------------------------------------- | ------------------------------------- |
| Stage 1 | 频域嵌入  | \(K_k(\nu) \times M(\nu)\) 复数乘法 + 零填充 | \(k_x \times k_y \to 128 \times 128\) |
| Stage 2 | 二维 IFFT | 行 FFT → 转置 → 列 FFT（含定点转换）         | \(128 \times 128\) 复数               |
| Stage 3 | 强度累加  | \(                                           | E                                     | ^2 \times \sigma_k\) 加权累加 | \(128 \times 128\) 浮点 |
| Stage 4 | 频移居中  | FFTshift 象限交换                            | \(128 \times 128\) 浮点               |
| Stage 5 | DDR 写回  | AXI-MM Burst 写                              | \(128 \times 128\) 浮点               |

在核迭代循环中，上述五个阶段以顺序执行方式串联。这种设计选择旨在保证 C/RTL 联合仿真的确定性延迟推断：第 \(k\) 个核的全部计算完成后，第 \(k+1\) 个核才开始执行，从而避免核间任务级重叠。

Stage 2（二维 IFFT）是整个流程的计算瓶颈，其内部采用 `DATAFLOW` 指令驱动的子流水线，实现“行变换 → 转置 → 列变换”的任务级流水。当某核的行 FFT 完成一行后，转置模块即可立即开始处理，无需等待全部行 FFT 完成。值得注意的是，二维 IFFT 的外层包装函数本身不启用 `DATAFLOW`，而是按“浮点→定点转换 → 2D FFT → 定点→浮点转换”的顺序依次调用。这种外层顺序调用模式确保了 FIFO 深度可精确推断，进一步提升了联合仿真的确定性。

#### 4.2.3 复数乘法处理单元

频域嵌入阶段的核心运算是复数乘法，其数学形式为 \((a + bi)(c + di) = (ac - bd) + (ad + bc)i\)。每次复数乘法需执行 4 次实数乘法和 2 次实数加法。在 HLS 实现中，该运算通过 `PIPELINE II=1` 指令实现每时钟周期完成一次复数乘法。

对于 golden_1024 配置（\(N_x = 8\)，\(k_x = 17\)），每个核需进行 \(17 \times 17 = 289\) 次复数乘法。设计中采用时分复用策略，由单个乘法单元在 289 个时钟周期内完成一个核的全部点乘操作。在资源允许的情况下，可进一步通过 `UNROLL` 指令并行化多个乘法器，以缩短嵌入阶段延迟。

### 4.3 关键功能模块的逻辑实现

#### 4.3.1 频域嵌入模块

频域嵌入模块实现本征核与掩模频谱的逐点复数乘法，并将结果零填充至 \(128 \times 128\) 的 FFT 网格。该模块作为 SOCS 流水线的入口阶段，其计算精度直接影响后续 IFFT 与强度累加的结果。

该模块的关键设计决策之一是采用编译时常量 `MAX_KERNEL_SIZE`（定义为 17，对应 \(N_x = 8\) 时的 \(2N_x + 1\)）作为循环边界，而非运行时变量 `kerX_actual` / `kerY_actual`。循环以 \(17 \times 17\) 的固定范围展开，在循环体内通过运行时条件判断仅对实际核覆盖区域执行计算，其余位置保持零值。这种策略确保了 Vitis HLS 联合仿真中循环迭代次数在综合阶段即可完全确定，从而避免了因运行时边界导致的仿真超时问题。

嵌入位置的计算需与 CPU 参考实现（Golden）保持一致的相对偏移，以确保 IFFT 后的空间域相位对齐。掩模频谱的中心坐标定义为 \(L_{xh} = L_x/2\)，核的局部频域窗口 \((k_x - N_x, k_y - N_y)\) 通过坐标偏移映射至全局频谱。边界检查机制进一步保证了核窗口不超出掩模频谱范围，越界位置自动置零。

上述设计在保证功能正确性的同时，显著提升了仿真效率和综合确定性，为后续模块的流水线化奠定了基础。

#### 4.3.2 二维 IFFT 模块设计

二维 IFFT 是 SOCS 计算流程中的核心模块，同时也是资源消耗最大的功能单元。本设计采用 Xilinx Vitis HLS 提供的 `hls::fft` IP 核，基于行-列分解法实现 \(128 \times 128\) 点二维逆变换。

为满足高精度与高吞吐率要求，设计通过参数化配置结构体对一维 FFT IP 核进行定制，主要配置参数如下：

- \(\log_2\) 变换长度 = 7（对应 128 点变换）；
- 输入/输出位宽采用 `ap_fixed<32,1>` 格式；
- 实现架构选择流水线流式 I/O（`pipelined_streaming_io`）；
- 缩放模式采用块浮点（`block_floating_point`）；
- 输出顺序为自然序（`natural_order`）。

二维 IFFT 的顶层包装函数按“浮点→定点转换 → 二维 FFT → 定点→浮点转换”的顺序依次调用，外层不启用 `DATAFLOW` 指令，以确保联合仿真中 FIFO 深度可精确推断。核心的二维变换则由内部函数通过 `#pragma HLS DATAFLOW` 驱动，实现“逐行 FFT → 矩阵转置 → 逐列 FFT”的任务级流水线。

行-列分解法的关键瓶颈在于矩阵转置。本设计利用 BRAM 双端口存储器实现隐式转置：行 FFT 结果按行写入临时缓冲区 `temp[128][128]`，列 FFT 按列读取，通过地址映射完成转置，无需额外专用转置缓冲区。该缓冲区声明为 `RAM_2P_BRAM`，配合 `PIPELINE II=1` 指令实现了每时钟周期处理一个元素的转置吞吐率。

块浮点缩放模式是本模块的另一关键设计选择。在该模式下，FFT IP 在每级蝶形运算后动态检测数据位宽，必要时进行自适应右移，并通过 `blk_exp` 输出累计移位量。最终的浮点补偿在定点-浮点转换阶段完成，补偿公式为：

$$
\text{output}_{\text{float}} = \text{output}_{\text{fixed}} \times 2^{\text{blk\_exp}}
$$

二维 IFFT 的总 `blk_exp` 为行变换与列变换累加值之和。选择块浮点而非逐级固定缩放（scaled）模式的原因在于：对于 \(\log_2 128 = 7\)（奇数）的变换长度，固定缩放模式在最后一级存在严格约束，容易导致精度退化；而块浮点模式通过动态溢出检测保留了更高的有效精度，同时避免了手动配置缩放因子的复杂性。

#### 4.3.3 强度累加模块

强度累加模块实现 SOCS 公式中 \(\sigma_k \times |E_k|^2\) 的加权累加操作，是整个流水线中唯一存在跨核数据依赖的模块——各核的强度贡献需累加至同一临时图像缓冲区 `tmpImg` 中。

该模块的核心计算为：

$$
\text{tmpImg}[y][x] \leftarrow \text{tmpImg}[y][x] + \sigma_k \times (|E_{\text{re}}|^2 + |E_{\text{im}}|^2)
$$

由于 IFFT 输出在定点-浮点转换阶段已完成块浮点缩放补偿，强度累加模块无需额外归一化处理，可直接执行平方与加权操作。该设计通过 `PIPELINE II=1` 指令实现了每时钟周期处理一个像素的高吞吐率。

上述实现已在 C 仿真与 CPU Golden 参考结果的对比验证中得到确认，端到端 RMSE 达到 \(2.93 \times 10^{-8}\)，充分满足了设计精度要求。


#### 4.3.4 FFTshift 与输出模块

FFTshift 模块用于将 IFFT 输出中的零频分量从频谱角落移动至中心位置，对应数学上的 \(\pi\) 相移操作。该模块采用基于模运算的源地址映射方式实现四个象限的对角交换（左上 \(\leftrightarrow\) 右下，右上 \(\leftrightarrow\) 左下）。通过 `PIPELINE II=1` 指令，该操作实现了每时钟周期处理一个像素的高吞吐率，每个像素仅需一次读操作和一次写操作。

输出模块则负责将 FFTshift 处理后的结果通过 AXI-MM 接口以突发模式写回 DDR 外部存储器。该模块同样采用 `PIPELINE II=1` 指令，确保了数据写回过程与前序计算阶段的无缝衔接。

上述两个模块共同构成了 SOCS 计算流水线的末端，确保最终的强度图像能够正确对齐并高效传输至主机侧。

### 4.4 存储系统优化与数据流控制

#### 4.4.1 AXI-MM 多端口 DDR 访问架构

为最大化 DDR 带宽利用率并避免总线竞争，本设计为 HLS IP 核配置了 7 个独立的 AXI-MM Master 接口，分别负责不同数据缓冲区的读写操作。这种多端口分离策略使得掩模频谱读取、本征核读取以及结果写回等操作能够在时间上实现重叠执行，从而显著提升了整体数据吞吐率。

| gmem1 | mskf_i（掩模频域虚部） | \(1024 \times 1024\)        | 1,048,576  | gmem1  | 只读     |

| gmem5 | tmpImg_ddr（中间结果） | \(128 \times 128\)          | 16,384     | gmem5  | 只写     |
| gmem6 | output（最终输出）     | \(128 \times 128\)          | 16,384     | gmem6  | 只写     |

各接口的突发传输参数（`latency`、`num_read_outstanding` 及 `max_read_burst_length`）均根据数据规模与访问模式进行了针对性优化。对于大规模掩模频谱接口，采用较长的突发长度以提升带宽利用率；而对于小规模特征值权重接口，则使用更为保守的参数以减少资源开销。该配置在保证数据一致性的同时，有效隐藏了 DDR 访问延迟。

#### 4.4.2 片上 BRAM 缓存架构

FPGA 片上 BRAM 用于存放计算过程中的中间数据，从而避免频繁的 DDR 访问并降低整体延迟。本设计将四个关键数组映射至 BRAM 资源：

**表 4-6 片上 BRAM 缓存分配**

| 数组         | 类型             | 尺寸               | 存储容量 | 用途              |
| ------------ | ---------------- | ------------------ | -------- | ----------------- |
| `fft_input`  | `complex<float>` | \(128 \times 128\) | 128 KB   | IFFT 输入缓冲     |
| `fft_output` | `complex<float>` | \(128 \times 128\) | 128 KB   | IFFT 输出缓冲     |
| `tmpImg`     | `float`          | \(128 \times 128\) | 64 KB    | 强度累加缓冲      |
| `tmpImgp`    | `float`          | \(128 \times 128\) | 64 KB    | FFTshift 输出缓冲 |

这些数组均通过 `#pragma HLS RESOURCE` 指令显式绑定为双端口 BRAM（`RAM_2P_BRAM`），支持同时读写操作以匹配流水线吞吐需求。

BRAM 资源优化的核心在于识别主要消耗源并采取针对性措施。FFT IP 实例是 BRAM 的主要占用者，通过将核间并行度从 2 降低至 1，并引入高效的 HLS FFT IP 核（含内部转置缓冲区与块浮点支持），最终 BRAM 占用为 399 个 BRAM_18K。基于实际综合验证器件 xcku3p（720 BRAM_18K）的占用率为 55%，部署至目标器件 xcku5p（960 BRAM_18K）时预估占用率为 42%，均远低于 75% 的设计上限。该优化经验表明：在资源受限的 FPGA 设计中，合理降低并行度并优先优化核心计算单元是平衡性能与资源利用率的最有效策略。

#### 4.4.3 AXI-Lite 运行时参数接口

除 AXI-MM 数据接口外，本设计还为 HLS IP 核配置了 AXI-Lite Slave 接口，用于运行时参数传递。该接口支持主机在运行时动态配置核数量 \(n_k\)、光学参数 \(N_x\)、\(N_y\) 以及掩模尺寸 \(L_x\)、\(L_y\)，从而使同一 IP 核能够适配不同配置场景，显著提升了架构的通用性与可扩展性。

固定 \(128 \times 128\) 的 FFT 网格尺寸，通过零填充机制支持 \(N_x\) 从 2 到 24 的广泛取值范围，实现了对多种核尺寸的灵活适配。

### 4.5 硬件优化技巧与精度控制

#### 4.5.1 块浮点 FFT 的精度控制

在 FPGA 上实现高精度 FFT 的主要挑战在于：浮点运算会消耗大量 DSP 资源，而定点运算则面临精度损失风险。本设计采用 `ap_fixed<32,1>` 格式（1 位符号位 + 31 位小数位）作为 FFT IP 的输入/输出数据类型，并配合块浮点缩放模式，在计算精度与资源利用率之间取得了良好平衡。

`ap_fixed<32,1>` 的量化步长约为 \(2^{-31} \approx 4.66 \times 10^{-10}\)，可表示的数值范围为 \([-1, 1)\)。SOCS 计算中各关键数据的典型范围均处于该定点格式的安全区间内，无溢出风险（见表 4-7）。

**表 4-7 SOCS 计算中主要数据的定点表示安全性分析**

| 数据类型  | 典型范围                        | 定点表示范围 | 精度        |
| --------- | ------------------------------- | ------------ | ----------- |
| 掩模频谱  | \([10^{-9}, 6 \times 10^{-2}]\) | \([-1, 1)\)  | 31-bit 小数 |
| 核系数    | \([0, 0.41]\)                   | \([-1, 1)\)  | 31-bit 小数 |
| 频域乘积  | \(\sim 0.025\)                  | \([-1, 1)\)  | 31-bit 小数 |
| IFFT 输出 | \([-0.1, 0.1]\)                 | \([-1, 1)\)  | 31-bit 小数 |

块浮点模式（`block_floating_point`）是本设计的关键选择。表 4-8 对比了块浮点模式与传统逐级缩放（scaled）模式的特性差异。

**表 4-8 块浮点模式与逐级缩放模式的对比**

| 特性     | 逐级缩放模式（scaled）                       | 块浮点模式（block_floating_point）       |
| -------- | -------------------------------------------- | ---------------------------------------- |
| 缩放策略 | 每级固定位移                                 | 每级动态检测溢出后自适应位移             |
| 精度保持 | 逐级截断，低位信息易丢失                     | 仅在检测到溢出时移位，保留更多有效位     |
| 参数约束 | \(n_{\text{fft}}\) 为奇数时末级必须为 0 或 1 | 无特殊约束                               |
| 输出补偿 | 无需补偿（固定缩放因子）                     | 需按 \(2^{\text{blk_exp}}\) 补偿         |
| 适用场景 | 固定 FFT 长度、偶数 \(n_{\text{fft}}\)       | 通用场景，奇偶 \(n_{\text{fft}}\) 均适用 |

由于本设计中 \(\log_2 128 = 7\) 为奇数，逐级缩放模式在最后一级存在严格约束，容易导致精度退化。块浮点模式通过 `blk_exp` 自适应跟踪累计移位量，无需手动配置缩放因子，且在相同位宽下保留了更高的有效精度。

端到端精度验证表明，本设计在 C 仿真及 CPU Golden 参考结果对比中，RMSE 达到 \(2.93 \times 10^{-8}\)，远优于 \(10^{-5}\) 的设计目标，充分满足了工业级光刻仿真精度要求。

#### 4.5.2 资源利用率优化

本设计的目标器件为 Xilinx Kintex UltraScale+ 系列 xcku5p-ffvb676-2-e，但实际综合验证使用的是资源规模较小的 xcku3p-ffvb676-2-e 器件。两者的主要差异在于逻辑资源容量：xcku5p 提供 960 个 BRAM_18K 和 1,824 个 DSP，而 xcku3p 仅提供 720 个 BRAM_18K 和 1,368 个 DSP。表 4-9 展示了基于 xcku3p 器件的实际资源利用情况，并给出了部署至 xcku5p 时的预估占用率。

**表 4-9 目标器件资源利用概况**

| 资源类型 | 使用量 | xcku3p 可用量 | xcku3p 占用率 | xcku5p 可用量 | xcku5p 占用率 |
| -------- | ------ | ------------- | ------------- | ------------- | ------------- |
| BRAM_18K | 399    | 720           | 55%           | 960           | 42%           |
| DSP      | 53     | 1,368         | 4%            | 1,824         | 3%            |
| FF       | 31,942 | 325,440       | 10%           | 433,920       | 7%            |
| LUT      | 37,098 | 162,720       | 23%           | 216,960       | 17%           |

DSP 资源的优化是本设计的重要成果之一。通过采用 HLS FFT IP 核，并将其内部蝶形运算与复数乘法配置为 LUT 实现（`use_luts`），DSP 消耗从早期直接 DFT 实现时的 8,064 个大幅降低至 53 个，在 xcku3p 上占用率仅为 4%，在 xcku5p 上更是降至 3%。该策略充分利用了 LUT 资源相对充裕的特点，在资源受限场景下实现了最优的面积-性能权衡。

此外，当前 LUT 占用率仅为 22%，仍存在一定剩余空间。为进一步提升可扩展性，未来可采用 LUTRAM 替代部分 BRAM 存储中间缓冲区，从而释放更多 BRAM 资源用于额外 FFT 实例或更高并行度设计。

通过上述精度控制与资源优化措施，本章提出的 FPGA 加速架构在保证高保真度的同时，实现了对 Kintex UltraScale+ 器件的良好适配，为后续板级验证与性能评估奠定了坚实基础。

#### 4.5.3 时钟性能与时序收敛

本设计以 300 MHz（周期 3.333 ns）为目标频率进行 HLS C 综合。时序分析报告显示，顶层模块存在 -0.79 ns 的负时序裕量（timing slack），表明 300 MHz 的目标频率尚未完全收敛。

违例路径主要集中在定点-浮点转换模块中。该模块涉及的浮点指数运算电路组合逻辑路径较长，成为关键路径；而二维 IFFT 等子模块单独综合时均满足时序约束，呈现正裕量。

为解决时序收敛问题，本设计采取以下优化策略：

1. **降低目标频率**：将综合目标调整至 280 MHz（周期 3.571 ns），负裕量缩小至约 -0.552 ns；进一步降至 250 MHz 可使裕量转为正值，但需在实际布局布线后进行验证。
2. **拆分关键运算**：将浮点指数运算替换为基于查找表的预计算实现，或将其拆分为多个流水级，从而缩短关键路径。
3. **插入流水线寄存器**：在浮点转换模块内部增加流水级，通过适当提高 `PIPELINE II` 值（例如 II=2），以少量吞吐率代价换取时序裕量的改善。

虽然当前综合结果存在时序违例，但 C 仿真与 C/RTL 联合仿真均已验证功能正确性。在后续 Vivado 物理综合与布局布线阶段，可通过寄存器重定时（register retiming）等后端优化手段进一步改善时序。初步估计，在 250–280 MHz 的频率范围内可实现可靠的时序收敛。
