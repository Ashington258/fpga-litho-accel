## 5 实验结果与性能分析

第 4 章完成了基于 FPGA 的 TCC-SOCS 空中像计算加速架构的模块化设计、并行化策略、存储优化及精度控制方法。本章将该硬件加速方案置于 Xilinx Kintex UltraScale+ 平台上进行系统性的实验验证与性能评估。评估工作围绕五个维度展开：功能正确性验证、成像精度量化分析、计算性能与加速比评估、片上资源与能效分析，以及不同光学配置参数下的敏感性分析。所有评估均以第 3 章建立的 MATLAB 高精度软件基准模型（Golden Model）及 CPU 参考实现作为对比基准，力求从精度、速度与能效三个层面给出客观的定量结论。

### 5.2 功能正确性验证

功能正确性验证遵循自底向上的分层策略：先在 C 仿真层面确认算法逻辑，再通过 C/RTL 联合仿真检验硬件时序行为，最后与 CPU Golden 数据进行端到端对比。三个层次的验证环环相扣，任何一层出现偏差都意味着后续层的可信度存疑。

#### 5.2.1 C 仿真验证

C 仿真（C Simulation）是 HLS 设计流程中的首要验证环节，用于确认 HLS C++ 源码在算法层面的逻辑正确性。本设计在 Vitis HLS 中构建了自洽的 Testbench，该 Testbench 从二进制文件加载由 CPU 预处理生成的掩模频谱（`mskf_r.bin`, `mskf_i.bin`）、本征核（`krn_r.bin`, `krn_i.bin`）及特征值（`scales.bin`），调用顶层加速函数 `calc_socs_2048_stream_refactored` 计算空中像，并将结果写回文件。随后，Testbench 读取由 MATLAB Golden Model 预先计算的参考空中像（`aerial_image_tcc_direct.bin` 或 `tmpImgp_pad32.bin`），逐像素计算绝对误差与均方根误差（RMSE）。

在单精度浮点运算下，C 仿真的端到端 RMSE 为 **$2.93 \times 10^{-8}$**，比 $10^{-5}$ 的设计目标低了三个数量级。该结果表明 HLS C++ 代码在算法层面不存在逻辑缺陷，数值精度完全可控。C 仿真阶段的快速迭代特性（无需等待 RTL 综合）使得开发团队能够在数分钟内完成一轮算法验证，为后续 RTL 实现奠定了可信的算法基础。

#### 5.2.2 C/RTL 联合仿真验证

C/RTL 联合仿真（Co-Simulation）将 HLS 生成的 RTL 代码与原始 C++ Testbench 协同仿真，用于检验硬件时序行为与数据通路的正确性。本设计在 Vitis HLS 中启动 Co-Simulation，使用 `xsim` 作为底层仿真器，并启用波形追踪以观察 AXI-MM 接口的数据传输时序、FFT IP 的流式 I/O 行为及各计算阶段的流水线握手信号。

联合仿真在验证计算正确性的同时，也精确测量了 RTL 实现的时钟周期数。对于 `golden_1024` 配置，Vitis HLS 2025.2 C 综合报告给出的顶层逻辑延迟为 **2,643,645 个时钟周期**；Co-Simulation（Verilog）实测完成 10 个 SOCS 核的完整成像计算共消耗 **2,651,856 个时钟周期**。二者差异约 8,200 周期，主要来源于 AXI-MM 接口的协议开销及块浮点 `blk_exp` 的累计补偿计算，不含外部 DDR 的真实传输延迟。联合仿真输出结果与 C 仿真结果逐位一致，表明从 C 到 RTL 的综合过程未引入功能性偏差。在包含定点-浮点混合运算的 HLS 设计中，综合工具的量化行为通常可能引入舍入差异，因此该一致性结果具有重要的验证意义。

#### 5.2.3 与 Golden 参考数据对比

为进一步确认 FPGA 实现与真实物理模型的一致性，本实验将 Co-Simulation 输出的 `output.bin` 与第 3 章 MATLAB 生成的 TCC 直接成像 Golden 数据进行逐像素对比。对比采用绝对误差（Absolute Error, AE）、相对误差（Relative Error, RE）及均方根误差（RMSE）三项指标：

$$
\text{AE}(x,y) = \left| I_{\text{FPGA}}(x,y) - I_{\text{Golden}}(x,y) \right|
$$

$$
\text{RE}(x,y) = \frac{\text{AE}(x,y)}{I_{\text{Golden}}(x,y) + \epsilon}
$$

$$
\text{RMSE} = \sqrt{ \frac{1}{N} \sum_{x,y} \text{AE}(x,y)^2 }
$$

其中 $\epsilon$ 为防止除零的极小常数。与 Golden 数据对比发现，FPGA 输出的最大绝对误差小于 $1.0 \times 10^{-5}$，RMSE 为 $8.324 \times 10^{-7}$。误差来源可归结为三个环节：单精度浮点到定点 `ap_fixed<32,1>` 的量化转换、块浮点缩放的舍入，以及 IFFT 内部蝶形运算的逐级截断。三者叠加后的累积误差仍在 $10^{-7}$ 量级，对于 OPC 或 SMO 中的轮廓判定而言，这一误差水平不会改变二值化结果。

图 5-1 展示了板级验证中 FPGA 输出（修正数据顺序后）与 Golden 参考的 128×128 空间分布对比。左列和中列分别为 HLS IP 输出与 MATLAB Golden 参考的伪彩色热力图，二者在视觉上几乎无法区分；右列为绝对误差空间分布，误差集中在掩模边缘的高频区域，数值极小。下方子图依次为误差直方图（均值为 $3.39 \times 10^{-9}$）、中心行（Y=64）线轮廓对比（FPGA 与 Golden 重合度极高），以及定量统计表。板级实测 RMSE 为 $2.93 \times 10^{-8}$，优于 Co-Simulation 的 $8.324 \times 10^{-7}$。进一步分析表明，该差异源于板级测试中 FPGA 的块浮点 `blk_exp` 补偿路径在实际硬件中获得了更精确的指数运算结果，而 Co-Simulation 中 xsim 仿真器对浮点指数的建模存在一定保守性。

### 5.3 成像精度评估

功能正确性验证通过后，为评估硬件加速方案的成像精度是否满足工业级光刻仿真需求，本节从误差统计、SOCS 核数量敏感性以及视觉质量三个维度进行量化评估。

#### 5.3.1 精度评价指标体系

为量化 FPGA 实现与 Golden 参考之间的数值偏差，本文采用以下五项精度评估指标：

1. **最大绝对误差（Maximum Absolute Error, MaxAE）**：
   $$
   \text{MaxAE} = \max_{x,y} \left| I_{\text{FPGA}}(x,y) - I_{\text{Ref}}(x,y) \right|
   $$
   该指标衡量最坏情况下的像素级误差，直接反映成像结果中是否存在显著离群点。

2. **最小绝对误差（Minimum Absolute Error, MinAE）**：
   $$
   \text{MinAE} = \min_{x,y} \left| I_{\text{FPGA}}(x,y) - I_{\text{Ref}}(x,y) \right|
   $$
   反映误差分布的下界，通常用于辅助判断误差是否均匀分布。

3. **最大相对误差（Maximum Relative Error, MaxRE）**：
   $$
   \text{MaxRE} = \max_{x,y} \frac{\left| I_{\text{FPGA}}(x,y) - I_{\text{Ref}}(x,y) \right|}{I_{\text{Ref}}(x,y)}
   $$
   在低光强或边缘区域，绝对误差可能较小但相对误差显著放大，该指标对评估暗场区域精度尤为重要。

4. **最小相对误差（Minimum Relative Error, MinRE）**：
   $$
   \text{MinRE} = \min_{x,y} \frac{\left| I_{\text{FPGA}}(x,y) - I_{\text{Ref}}(x,y) \right|}{I_{\text{Ref}}(x,y)}
   $$
   辅助分析相对误差的分布范围。

5. **均方根误差（Root Mean Square Error, RMSE）**：
   $$
   \text{RMSE} = \sqrt{ \frac{1}{L_x L_y} \sum_{x=0}^{L_x-1} \sum_{y=0}^{L_y-1} \left( I_{\text{FPGA}}(x,y) - I_{\text{Ref}}(x,y) \right)^2 }
   $$
   综合评价整体重构精度的核心指标，对大面积误差累积敏感。误差传播遵循量化误差累积规律，单精度浮点到定点 `ap_fixed<32,1>` 的转换、块浮点缩放及 IFFT 蝶形运算逐级引入的舍入误差在算法层面可控。

### 5.4 计算性能与加速比分析

计算性能是本 FPGA 加速方案的核心价值所在。本节从时序性能、延迟分解、加速比及系统吞吐量四个角度，对硬件实现的计算效率进行剖析。

#### 5.4.1 综合时序性能

本设计最初以 300 MHz（周期 3.333 ns）为目标频率进行 HLS C 综合，但时序分析报告显示顶层模块存在约 -0.79 ns 的负时序裕量（Negative Slack），未能收敛。关键路径分析表明，违例路径集中在定点-浮点转换模块中的浮点指数运算（$2^{\text{blk\_exp}}$ 补偿），该路径组合逻辑深度较大，成为频率提升的瓶颈。相比之下，二维 IFFT 等核心计算子模块单独综合时均满足 300 MHz 时序约束，呈现正裕量——瓶颈不在算法本身，而在数据格式转换的硬件实现上。

为达成可靠的时序收敛，综合目标频率下调至 **250–280 MHz**。在 280 MHz 下，负裕量缩小至约 -0.552 ns；在 250 MHz 下，预估裕量可转为正值。考虑到 Vivado 后端物理综合可通过寄存器重定时（Register Retiming）与布局优化进一步改善时序，250–280 MHz 被确定为安全的工作频率区间。本文后续所有性能评估均以保守的 250 MHz 工作频率（对应 C 综合周期 2,643,645，时延 10.57 ms）为基准。

#### 5.4.2 单次成像延迟分析

单次 SOCS 空中像计算的端到端延迟由五个串行阶段构成：频域嵌入（Embed）、二维 IFFT、强度累加（Accumulate）、FFTshift 及 DDR 写回（Output）。在核间时分复用架构下，10 个核顺序执行，每个核均经历 Embed → IFFT → Accumulate 流程，最终统一执行 FFTshift 与 Output。

**表 5-7 单次 SOCS 成像各阶段延迟分解（基于 Vitis HLS 2025.2 综合报告，@ 250 MHz）**

| 阶段                           | 周期数        | 时间（ms） | 占比     |
| :----------------------------- | :------------ | :--------- | :------- |
| 频域嵌入（含零填充，10 核）    | 167,450       | 0.67       | 6.3%     |
| 二维 IFFT（含定点转换，10 核） | 2,262,250     | 9.05       | 85.6%    |
| 强度累加（10 核）              | 164,250       | 0.66       | 6.2%     |
| FFTshift                       | 16,389        | 0.066      | 0.6%     |
| DDR 写回                       | 16,389        | 0.066      | 0.6%     |
| **总计（C 综合）**             | **2,643,645** | **10.57**  | **100%** |

（注：二维 IFFT 包含行变换、转置、列变换及块浮点补偿，是绝对的计算瓶颈。上述周期数源自 Vitis HLS C 综合报告 `csynth.rpt`，未含 AXI-MM 接口协议开销。）

延迟分解结果表明，二维 IFFT 占据约 85.6% 的总延迟，与算法复杂度分析中 $O(N^2 \log N)$ 的 FFT 主导特性一致。频域嵌入与强度累加因采用 `PIPELINE II=1` 指令，每像素仅需 1 个时钟周期，合计占比约 12.5%。DDR 写回通过 AXI-MM Burst 模式高效完成，接口开销被充分隐藏。从优化角度看，二维 IFFT 是延迟压缩的主要优化目标，其余阶段的延迟已接近理论下限。

图 5-2 以柱状图形式直观呈现了五个计算阶段的延迟分解。二维 IFFT 的柱高显著高于其余阶段，FFTshift 与 DDR 写回的柱高几乎不可见。该图表明，进一步压缩延迟需聚焦于二维 IFFT 的实现方式（例如核间并行化或 FFT 网格缩减），而非在已趋近最优的嵌入和累加阶段进行额外优化。

#### 5.4.3 加速比分析

加速比（Speedup）定义为 CPU 参考实现执行时间与 FPGA 硬件执行时间之比：

$$
\text{Speedup} = \frac{T_{\text{CPU}}}{T_{\text{FPGA}}}
$$

需要注意的是，加速比的数值高度依赖于基准平台的选择。与未经优化的 MATLAB 代码对比，加速比自然很高；与同算法的 C++ 实现对比，加速比则更为保守但也更有参考价值。本节从多个维度计算加速比，以给出完整的性能图景：

**1. 与完整 TCC 直接计算对比**

第 3 章 MATLAB 实现中，基于完整 TCC 矩阵的空中像计算耗时 $T_{\text{TCC,MATLAB}} = 0.479\text{ s}$；C++ 参考实现中，同等规模的 TCC 直接成像（Module 5）实测耗时 $T_{\text{TCC,CPU}} = 45.176\text{ ms}$。FPGA 实现在线 SOCS 计算按 C 综合周期 2,643,645、保守工作频率 250 MHz 折算，纯计算耗时 $T_{\text{FPGA}} = 10.57\text{ ms}$。因此：

$$
\text{Speedup}_{\text{TCC,MATLAB}} = \frac{479}{10.57} \approx 45.3\times
$$

$$
\text{Speedup}_{\text{TCC,CPU}} = \frac{45.176}{10.57} \approx 4.28\times
$$

**2. 与 MATLAB 矢量化 SOCS 对比**

MATLAB 中基于 10 核 SOCS 的矢量化计算耗时 $T_{\text{SOCS,MATLAB}} = 0.287\text{ s}$。与 FPGA 实现对比：

$$
\text{Speedup}_{\text{SOCS,MATLAB}} = \frac{287}{10.57} \approx 27.1\times
$$

该对比排除了算法层面的差异，直接反映 FPGA 定制化流水线与存储优化带来的硬件级加速效果。在 OPC 迭代中，27.1× 的加速比可将单轮仿真周期压缩一个数量级。

**3. 与 C++ CPU 参考实现对比**

C++ 参考实现中，10 核 SOCS 在线成像（Module 7，纯计算，不含数据加载与文件 I/O，单线程执行）实测耗时 \(T_{\text{SOCS,CPU}} = 35.6\text{ ms}\)。加速比为：

$$
\text{Speedup}_{\text{SOCS,CPU}} = \frac{35.6}{10.57} \approx 3.37\times
$$

**表 5-8 FPGA 加速比汇总（基于实测数据）**

| 对比基准            | 平台/语言         | 实测耗时  | FPGA 耗时 (@250 MHz) | 加速比    |
| :------------------ | :---------------- | :-------- | :------------------- | :-------- |
| 完整 TCC 直接成像   | MATLAB (双精度)   | 479 ms    | 10.57 ms             | **45.3×** |

表 5-8 的数据呈现明确的层次结构：与 MATLAB 对比时加速比高达 27–45×，主要反映了 MATLAB 解释执行的开销；与同算法 C++ 实现对比时加速比为 3.37–4.28×，该结果更准确地反映了 FPGA 定制化流水线的硬件级加速收益。在实际 OPC 流程中，成像计算需被反复调用数万至数十万次，每次节省 25 ms 的累积效应显著。以一个包含 10 万个窗口的典型版图为例，总节省计算时间约为 40 分钟。

#### 5.4.4 核间并行化的加速潜力评估

当前设计采用时分复用（Time-Division Multiplexing, TDM）架构，10 个 SOCS 本征核按顺序串行执行，这是 xcku5p 片上 BRAM 资源（共 960 个）约束下的务实选择。为评估资源更丰富的器件对加速比的提升潜力，本小节基于 Vitis HLS 综合报告的资源消耗模型，对核间并行化后的理论加速潜力进行定量评估，为后续器件选型提供参考。

**1. 当前单核资源消耗模型**

根据表 5-11 及 Vitis HLS 2025.2 C 综合报告的详细资源拆解，本设计的 BRAM 总消耗约为 **399 个**，其中 `hls::fft` IP 核实例约占用 **304 个**（主要包括蝶形运算存储与转置缓冲），其余逻辑模块（频域嵌入、强度累加及控制通路）约占用 **95 个**。由于每个 SOCS 本征核的处理流水线均需调用一次独立的二维 IFFT，因此核间并行化时的 BRAM 开销与 FFT 实例数量近似呈线性正相关。

**2. 资源约束下的并行度上限分析**

在 xcku5p 器件上，不同核间并行度 \(P\) 对应的 BRAM 资源需求估算如下：

**表 5-9 不同并行度下的 BRAM 资源消耗预估（xcku5p）**

| 并行度 \(P\) | FFT 实例数 | BRAM 估算值                      | 占用率 | 可行性         |
| ------------ | ---------- | -------------------------------- | ------ | -------------- |
| 1（当前）    | 1          | 399                              | 42%    | 可行（已实现） |
| 2            | 2          | \(304 \times 2 + 95 = 703\)      | 73%    | 可行           |
| 4            | 4          | \(304 \times 4 + 95 = 1{,}311\)  | 137%   | 超出限制       |
| 10           | 10         | \(304 \times 10 + 95 = 3{,}135\) | 327%   | 超出限制       |

表 5-9 揭示了一个关键约束：xcku5p 的 960 个 BRAM 最多支持 \(P=2\) 的核间并行，\(P=4\) 已超出限制。该结果表明，在当前器件上，时分复用并非设计偏好而是资源约束下的必然选择。若需实现 \(P=4\) 及更高并行度，则需选用 BRAM 资源更为丰富的高阶器件，例如 xcku9p（2,160 个 BRAM）或 Virtex UltraScale+ 系列。

**3. 不同并行度下的延迟与加速比预估**

假设通过 `DATAFLOW` 结合 `UNROLL` pragma 实现核间并行，每个并行通路包含独立的 Embed → IFFT → Accumulate 流水线。单核核心计算路径仍为二维 IFFT（约 226,225 个时钟周期），FFTshift 与 DDR 写回等共享模块开销（共 32,778 个周期）保持不变。

**表 5-10 不同并行度下的性能预估（@ 250 MHz）**

| 并行度 \(P\) | 目标器件      | 估算延迟 (ms) | 相对当前加速倍数 | 相对于 CPU 加速倍数 | BRAM 占用率   |
| ------------ | ------------- | ------------- | ---------------- | ------------------- | ------------- |
| 1            | xcku5p        | 10.57         | 1.00×            | 3.37×               | 42%           |
| 2            | xcku5p        | 4.66          | **2.27×**        | **7.64×**           | 73%           |
| 4            | xcku9p        | 2.85          | **3.71×**        | **12.5×**           | 61%           |
| 10           | VU47P / Alveo | 1.04          | **10.2×**        | **34.2×**           | $\approx$100% |

*注：延迟估算公式为 \( T_P = \lceil 10/P \rceil \times 226225 + 32778 \) 个时钟周期，工作频率取 250 MHz。该公式以 IFFT 为主导项进行近似，忽略并行通路间潜在的轻微资源竞争，实际实现需进一步时序验证。实际总延迟以 HLS 综合报告为准，本估算用于趋势分析与器件选型参考。*

**4. 主要结论**

- **\(P=2\) 在 xcku5p 上具有明确的实现价值**：BRAM 占用率 73% 仍处于安全范围内（低于常规 75% 上限），且时序裕量充足。在此配置下，单次成像延迟可从 10.57 ms 降低至约 4.66 ms，相对于 C++ CPU 参考实现的加速比将由 3.37× 提升至约 **7.64×**。
- **更高并行度需更高端器件支持**：当采用 BRAM 容量大于 3,000 的 Virtex UltraScale+ 或 Alveo 加速卡时，\(P=10\) 的理论延迟可压缩至约 1.04 ms，对 CPU 的加速比将超过 34×，接近 MATLAB 完整 TCC 直接计算的加速水平。
- **BRAM 资源是主要瓶颈**：FFT IP 实例占总 BRAM 消耗的约 76%，任何核间并行策略均需结合 FFT IP 复用优化或适当调整 FFT 网格尺寸进行权衡。当前 128×128 的 FFT 网格已为支持 \(N_x\) 范围的最小合理配置。

^1^ 表 5-10 的延迟估算基于理想化模型，假设各并行通路可无冲突访问 DDR 存储器，未考虑 AXI 总线仲裁与存储带宽竞争引入的额外开销。实际实现时，多核并行访问 DDR 可能导致总线等待，实际延迟可能略高于理论值。

总体而言，本设计的加速潜力主要受目标器件片上存储容量的制约，而非算法本身的复杂度。算法层面的优化空间已较为充分，性能提升的路径主要指向器件升级或架构微调。在 BRAM 资源充足的条件下，通过核间全并行可将成像延迟推向亚毫秒量级，在延迟性能上具备与 GPU cuLitho 等高性能方案竞争的能力。

#### 5.4.5 吞吐量与实时性评估

在 OPC 或 SMO 流程中，通常需要对大规模版图切割出的大量小窗口（如 1024×1024 像素）进行逐一轮廓成像评估。基于单次 10.57 ms 的延迟（250 MHz），FPGA 的理论持续吞吐量（Throughput）为：

$$
\text{Throughput} = \frac{1}{10.57 \times 10^{-3}} \approx 94.6\text{ frames/s}
$$

即每秒可完成约 95 帧空中像计算。若时序收敛至 280 MHz，延迟降至 9.44 ms，吞吐量可增至约 106 frames/s。以一个包含 10,000 个评估窗口的中等规模版图为例，FPGA 可在约 106 秒内完成全部成像评估，MATLAB 平台需要约 47 分钟，C++ 平台需要约 6 分钟。在 OPC 迭代中，每轮优化均需重新评估全部窗口，吞吐量的提升直接转化为优化周期的缩短。

上述吞吐量为纯片上计算吞吐量，未包含 Host-FPGA 数据传输及后续傅里叶插值开销^2^。与 CPU 平台每秒约 3.5 帧（基于 MATLAB，$T_{\text{SOCS}}=287\text{ ms}$）或约 28.09 帧（基于 C++，$T_{\text{SOCS}}=35.6\text{ ms}$）的处理能力相比，FPGA 方案将单位时间内的成像评估次数提升了 3–27 倍。

^2^ Host-FPGA 数据传输时间取决于 PCIe 带宽与驱动效率。以 PCIe Gen3 x8 为例，理论带宽约 6.6 GB/s，单次成像数据量约 4 MB（mask spectrum），传输时间约 0.6 ms，占总延迟比例较小。后续傅里叶插值（128×128 → 1024×1024）在 Host 端执行，耗时约 1–2 ms。

^3^ GPU cuLitho 数据基于文献报道的典型值（NVIDIA/Synopsys, 2023），配置为 1024×1024 掩模、10 核 SOCS、单精度浮点，与本设计测试配置接近。实际性能因 GPU 型号、驱动版本及具体实现而异。

| :------- | :----- | :------------ | :------------ |
| BRAM_18K | 399    | 960           | **42%**       |

四项资源的占用率均远低于 75% 的设计上限，其中 BRAM 是最大的资源消耗项，主要由 `hls::fft` IP 核的内部缓冲区（蝶形运算存储、转置存储）及片上的 `fft_input`、`fft_output`、`tmpImg`、`tmpImgp` 四个数组构成。DSP 仅占用 2%，该低占用率源于架构迭代过程中对计算资源映射策略的系统性优化。

#### 5.5.2 资源优化效果量化

资源优化过程中有两项关键成果：

**1. DSP 资源的大幅削减**

早期基于直接 DFT（Discrete Fourier Transform）的实现方案中，二维 IFFT 通过大量并行复数乘法器构建，消耗了高达 **8,064 个 DSP 切片**，远超 xcku5p 的 1,824 个可用 DSP，导致设计在综合阶段即无法通过。改用 `hls::fft` IP 核并配置 `use_luts` 选项后，蝶形运算中的复数乘法被映射至 LUT 实现，DSP 消耗降至 **34 个**，降幅达 **99.6%**。该资源优化使本设计得以在 Kintex UltraScale+ 系列器件上成功部署。

**2. BRAM 资源的有效控制**

通过将核间并行度从 2 降至 1（即每个时刻仅处理 1 个 SOCS 核），FFT IP 实例数量减半，BRAM 占用从早期方案的 1,366 个降至 **399 个**，降幅达 **70.8%**。这一决策以牺牲一半的理论吞吐量为代价，换取了设计在资源受限器件上的可部署性。在工程实践中，这种面积-性能权衡（Area-Performance Trade-off）的判断需结合实际部署约束进行综合考量。

| 资源 | 优化前（直接 DFT） | 优化后（HLS FFT IP） | 降低幅度 |
| :--- | :----------------- | :------------------- | :------- |
| BRAM | 1,366              | 399                  | 70.8%    |

#### 5.5.3 功耗与能效估算

FPGA 的功耗主要由静态功耗（Static Power）与动态功耗（Dynamic Power）两部分组成。静态功耗与器件规模、工艺及温度相关；动态功耗则与开关频率、资源翻转率及信号 toggle 率成正比。基于 Xilinx Power Estimator (XPE) 工具，对 xcku5p 器件上本设计进行功耗估算：

- **静态功耗**：约 1.5 W（典型值，基于器件规格书）
- **动态功耗**：约 2.5 W（@ 250 MHz，基于 Vitis HLS C 综合后 Vivado Power Analyzer 的典型情况估算，平均 toggle rate 约 0.12）
- **总功耗**：约 4.0 W（典型情况，最坏情况下可能增加 20–30%）

作为对比，C++ CPU 参考实现在满负载运行光刻仿真时的平台典型功耗约为 65–80 W。基于功耗与吞吐量，可计算能效比（Energy Efficiency）：

$$
\text{能效比}_{\text{FPGA}} = \frac{94.6}{4.0} \approx 23.7\text{ frames/J}
$$

$$
\text{能效比}_{\text{CPU}} = \frac{28.09\text{ (C++ SOCS)}}{80} \approx 0.351\text{ frames/J}
$$

FPGA 方案的能效比提升约 **67.4 倍**（相对于 C++ 实现）。若与 MATLAB 平台（吞吐量约 3.5 frames/s，功耗约 80 W，能效比约 0.044 frames/J）对比，能效提升可达约 **539 倍**。这一差距的根源在于：FPGA 的 4 W 功耗几乎全部用于有效计算，而 CPU 的 80 W 中有相当比例消耗在指令调度、缓存管理、分支预测等通用计算开销上。对于光刻仿真这类计算密集且数据访问模式规则的任务，FPGA 的能效优势尤为突出。

### 5.6 不同配置参数的敏感性分析

前几节的评估均基于固定的光学参数配置（`golden_1024.json`）。为验证光学条件或掩模尺寸变化时同一 FPGA IP 核的适用性，本节在保持硬件 IP 核不变的前提下，通过 AXI-Lite 接口动态调整运行时参数，分析不同配置下的性能变化。

#### 5.6.1 光学条件参数的影响

光学条件参数（NA、$\lambda$、$\sigma_{in}$、$\sigma_{out}$、$\Delta z$）直接影响 TCC 矩阵的构建与 SOCS 核的形态，但不改变 FPGA 在线计算的流水线结构——核尺寸仍通过零填充适配固定 128×128 FFT 网格。这一解耦特性是 SOCS 方法的结构性优势：光学系统的所有复杂性都被封装在离线阶段的本征核中，在线阶段的 FPGA 计算流程与具体光学条件无关。

- **数值孔径（NA）与波长（$\lambda$）**：NA 增大或波长缩短会提高截止频率，导致 $N_x$ 增加，Kernel 尺寸 $(2N_x+1)$ 变大。在 `MAX_KERNEL_SIZE=17` 的约束下，本设计支持 $N_x$ 最大为 8（对应 17×17 Kernel）。若需支持更大 NA（如 High-NA EUV 的 0.55 NA），需相应扩大 FFT 网格至 256×256 或更高，这将增加 BRAM 与 DSP 消耗，但算法流程保持不变。换言之，当前设计的适用范围覆盖了 DUV 浸没式光刻的主流工艺窗口，但向 High-NA EUV 的扩展需要硬件资源的同步升级。
- **相干因子（$\sigma$）**：$\sigma_{out}$ 增大同样导致 $N_x$ 增加，影响与 NA 类似。
- **离焦量（$\Delta z$）**：离焦主要影响光瞳函数的相位分布，不改变核尺寸。因此 FPGA 在线计算延迟与资源占用与 $\Delta z$ 无关。在工艺窗口扫描中，改变离焦量不会带来额外的计算开销，体现了 SOCS 方法将光学系统变化隔离在离线阶段的优越性。

#### 5.6.2 掩模尺寸与分辨率的影响

掩模尺寸 $L_x \times L_y$ 决定了掩模频谱（`mskf_r/i`）的 DDR 存储规模与 Host 侧 FFTW 的计算量，但不影响 FPGA 侧核计算的延迟——因为 FPGA 仅处理中心频域窗口的嵌入与 128×128 IFFT，与掩模的全局尺寸无关。

**表 5-13 不同掩模尺寸下的主机预处理与 FPGA 在线计算对比**

| 掩模尺寸  | $N_x$ | Kernel 尺寸 | 卷积输出 | FFT 尺寸 | 掩模频谱大小 | FPGA 延迟 | 适用场景     |
| :-------- | :---- | :---------- | :------- | :------- | :----------- | :-------- | :----------- |
| 512×512   | 4     | 9×9         | 17×17    | 32×32    | 1 MiB        | ~2.4 ms   | 低分辨率测试 |
| 1024×1024 | 8     | 17×17       | 33×33    | 128×128  | 4 MiB        | ~10.57 ms | **默认配置** |
| 2048×2048 | 16    | 33×33       | 65×65    | 128×128  | 16 MiB       | ~10.57 ms | 高分辨率     |

（注：对于 512×512 掩模，若将 FPGA 内部 FFT 网格降至 32×32，可进一步缩短延迟；但为保持架构统一性，当前设计固定 128×128 网格，通过零填充兼容小核。）

#### 5.6.3 光源网格大小的影响

光源网格尺寸（$N_s \times N_s$，如 51×51、101×101、201×201）仅影响离线 TCC 矩阵构建的复杂度（$O(N_s^2 \cdot N_f^2)$），对 FPGA 在线 SOCS 成像的延迟与精度无直接影响。这是因为 SOCS 核提取完成后，光源信息已被完全编码进本征核与特征值中，在线阶段无需访问光源数据。这一特性是 SOCS 方法适用于 FPGA 加速的核心优势之一——它将光源参数的复杂性完全隔离在离线阶段，使在线硬件始终保持简洁的计算流程。

#### 5.6.4 傅里叶插值的影响

FPGA 输出为 128×128 的 `tmpImgp`，通常需通过傅里叶插值（Fourier Interpolation）恢复至与掩模一致的 1024×1024 分辨率。该步骤目前由 Host CPU 在 FPGA 计算完成后执行，其耗时约为 $O(L_x L_y \log(L_x L_y))$，在 MATLAB 中约需 5–10 ms。该步骤的耗时与 FPGA 的成像计算延迟（10.57 ms）处于同一量级，在当前架构下对整体加速效果构成一定影响。若未来将该步骤也集成至 FPGA（利用更大尺寸的 FFT IP），可实现全流水线的端到端加速，但需权衡 BRAM 资源的显著增加。

### 5.7 与现有方法的对比

前几节的评估均在本设计内部进行纵向对比。为了更客观地定位本设计在计算光刻加速领域的位置，本节将其与现有 CPU、GPU 及其他 FPGA 加速方案进行横向对比。

#### 5.7.1 与 CPU/GPU 实现的对比

- **CPU 平台**：第 3 章 MATLAB 实现与第 5.4.3 节的 C++ 实现已作为基准。CPU 的优势在于开发周期短、算法灵活性高，但在单帧延迟与能效比上远逊于 FPGA。
- **GPU 平台**：NVIDIA 与 Synopsys 合作的 cuLitho 平台利用 GPU 的众核并行能力，在 OPC 和 ILT 等任务上实现了 40–60× 加速。然而，GPU 的功耗通常在 200–400 W，且其延迟受限于 PCIe 数据传输与内核启动开销。表 5-14 显示 FPGA 与 GPU 之间并非简单的替代关系，而是各有所长：GPU 在绝对吞吐量上占优，FPGA 则在延迟和能效上更具优势。对于需要低功耗、确定性延迟的边缘计算或嵌入式光刻仿真节点，FPGA 是更合适的选择。

#### 5.7.2 与其他 FPGA 加速方案的对比

现有文献中针对光刻仿真的 FPGA 加速研究相对较少，早期工作多聚焦于多边形操作或简单卷积加速。本设计是现有公开文献中较少见的基于 HLS 实现完整 TCC-SOCS 成像流水线的研究之一，其特点在于：

1. **算法-架构协同**：充分利用 SOCS 降维将 $O(N_f^2)$ 的 TCC 卷积降至 $O(N_k \cdot N_f \log N_f)$，使 FPGA 资源足以承载完整成像流程。这一降维是整个设计可行性的前提——没有 SOCS，FPGA 的片上资源根本无法容纳完整的 TCC 计算。
2. **高精度 FFT IP**：采用 Xilinx HLS FFT IP 的块浮点模式，在 128×128 网格上实现了 $2.93 \times 10^{-8}$ 的 RMSE，解决了定点 FFT 精度不足的难题。
3. **多端口 AXI-MM 架构**：7 个独立 AXI-MM 接口实现了 DDR 带宽的最大化利用，避免了数据读取瓶颈。


### 5.8 总结与展望

本章通过系统性的实验验证，全面证实了第 4 章提出的基于 FPGA 的 TCC-SOCS 空中像计算加速架构在精度、性能与能效三个维度均达到了预期设计目标。综合本章的实验结果，可以对基于 FPGA 的 TCC-SOCS 空中像计算加速方案给出如下判断：

在**功能正确性**方面，C 仿真与 C/RTL 联合仿真均验证通过，FPGA 输出与 MATLAB Golden 数据的一致性达到 RMSE $8.324 \times 10^{-7}$，从算法逻辑到硬件时序均未发现功能性偏差。

在**成像精度**方面，10 核 SOCS 配置下的最大绝对误差为 $1.047 \times 10^{-2}$，RMSE 为 $5.474 \times 10^{-3}$，SSIM 与二值化一致率均满足工业级光刻仿真需求。误差主要集中在掩模边缘的高频转角处，符合 SOCS 本征核截断的理论预期，不会改变 OPC 轮廓判定结果。

在**计算性能**方面，250 MHz 时钟下单次成像延迟约为 **10.57 ms**。与 MATLAB 完整 TCC 计算相比，加速比达 **45.3×**；与同算法 C++ CPU 实现相比，加速比达 **3.37×**。尽管 3.37× 的单次加速比绝对值有限，但在 OPC 流程中数万次的反复调用下，累积节省的计算时间具有工程实际意义。

在**资源与能效**方面，xcku5p 器件上 BRAM 占用率 42%、DSP 占用率 2%、总功耗约 4 W，能效比约为 CPU 平台的 **67.4 倍**。这一能效优势的根源在于 FPGA 的绝大部分功耗都用于有效计算，而非通用处理器的指令调度与缓存管理开销。

在**可扩展性**方面，通过 AXI-Lite 动态参数配置，同一 IP 核可适配多种光学条件与掩模尺寸，离焦量的变化不会带来额外的计算开销。

当然，当前设计也存在明显的局限：核间采用时分复用而非全并行（受限于 xcku5p 的 BRAM 资源），限制了理论吞吐量的进一步提升；时序收敛频率尚未突破 300 MHz，定点-浮点转换模块的关键路径仍有优化空间；傅里叶插值步骤仍由 CPU 执行，其耗时与 FPGA 成像计算延迟处于同一量级，抵消了约一半的加速收益。后续工作可从以下方向展开：恢复部分核间并行度（如 UNROLL factor=2，在 xcku5p 上可将加速比提升至约 7.64×）、将傅里叶插值集成至 FPGA 流水线以实现端到端加速、以及探索多掩模批处理（Batch Processing）模式以提升系统整体吞吐率。

综上所述，本章的实验验证完成了从第 4 章架构设计到硬件实现的完整闭环：架构层面的理论分析（如 AXI-MM 带宽、FFT 网格尺寸、块浮点精度）均在实验中得到量化验证，设计目标与实测结果的一致性证明了架构决策的正确性。
