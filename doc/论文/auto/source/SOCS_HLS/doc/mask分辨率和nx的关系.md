### 参数派生关系

```
Nx (频域采样点数)
  │
  ├──→ kerX = 2×Nx + 1  (Kernel宽度)
  │     例: Nx=4 → kerX=9
  │
  ├──→ convX = 4×Nx + 1  (卷积输出宽度)
  │     例: Nx=4 → convX=17
  │
  └──→ fftConvX = next_pow2(convX)  (FFT执行尺寸)
        例: convX=17 → fftConvX=32
        例: convX=65 → fftConvX=128
```

### 资源估算公式

#### DSP消耗公式

基于Direct DFT实现（当前方案）：

$$\text{DSP} \approx 84 \times \frac{N}{32} + 3 \times \frac{\text{kerX}}{9} + 26$$

- **84 DSP**：fft_2d模块基线（32×32 FFT）
- **3 DSP**：复数乘法器（kernel嵌入）
- **26 DSP**：sin_or_cos模块（2×13）

#### BRAM消耗公式

BRAM主要用于存储中间数组（尺寸随FFT点数平方增长）：

$$\text{BRAM} \approx 648 \times \left(\frac{N}{32}\right)^2 + 146 \times \frac{\text{kerX}^2}{81}$$

- **648 BRAM**：fft_2d中间缓冲（temp1/temp2/temp3）
- **146 BRAM**：kernel存储及其他

#### FF/LUT消耗公式

近似线性增长：

$$\text{FF} \approx 85,970 \times \frac{N}{32}$$

$$\text{LUT} \approx 124,646 \times \frac{N}{32}$$

#### ✅ DSP资源充足（所有配置）

- 最大配置DSP需求：385（仅占21.2%）
- DSP不是瓶颈，Direct DFT实现可接受

| FFT尺寸 | BRAM需求范围  | 占用率   | 状态        |

### 资源优化方案

#### 方案1：DDR缓存替代BRAM（强烈推荐）

将大型中间数组（temp1/temp2/temp3）从BRAM改为AXI-MM DDR访问：

```cpp
// 当前：BRAM存储（BRAM消耗高）
#pragma HLS RESOURCE variable=temp1 core=RAM_2P_BRAM

// 改为：DDR访问（BRAM降至~200）
#pragma HLS INTERFACE m_axi port=temp1_ddr offset=slave bundle=gmem6
```

**预期效果**：
- BRAM从10,608降至~200（仅保留小尺寸buffer）
- 新增1个AXI-MM接口（DDR带宽需求增加）
- Latency略有增加（DDR访问延迟）

#### 方案2：HLS FFT IP替换Direct DFT

使用Vitis HLS FFT IP（`hls::fft`）可显著降低DSP和BRAM：

| 实现方式           | 128×128 DSP | 128×128 BRAM |
| ------------------ | ----------- | ------------ |
| Direct DFT（当前） | 365         | 10,608       |
| HLS FFT IP         | ~200        | ~1,500       |
| **优化效果**       | **-45%**    | **-86%**     |

**参考实现**：`reference/vitis_hls_ftt的实现/interface_stream/`

#### 方案3：分块计算（降低峰值资源）

将大FFT分解为多次小FFT（如128×128 → 4次64×64）：

- BRAM峰值降至64×64级别（~2,719）
- Latency增加4倍
- 适用于对实时性要求不高的场景

### 推荐配置选择

| 应用场景         | 推荐配置            | BRAM状态 | 优化需求           |
| ---------------- | ------------------- | -------- | ------------------ |
| 低分辨率测试     | Nx=2, FFT=16×16     | ✅ 17%    | 无需优化           |
| 当前项目验证     | Nx=4, FFT=32×32     | ⚠️ 82%    | 可接受，无需优化   |
| 中等分辨率扩展   | Nx=6-8, FFT=32-64   | ⚠️-❌      | **必须DDR缓存**    |
| 高分辨率生产部署 | Nx=16+, FFT=128×128 | ❌        | **必须DDR+FFT IP** |

---

### 关键结论

1. **DSP不是瓶颈**：所有配置DSP占用≤21%，资源充足
2. **BRAM是主要瓶颈**：64×64 FFT以上必须优化
3. **推荐优化策略**：DDR缓存 + HLS FFT IP组合方案
4. **优先级**：先实现DDR缓存（解决BRAM），再集成FFT IP（进一步优化DSP/BRAM）
